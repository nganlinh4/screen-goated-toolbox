# Manage Model Catalog

Safely add, edit, or remove models in screen-goated-toolbox.

## Inputs

Ask the user for any that were not provided:

1. **Operation** — `add`, `edit`, or `remove`.
2. **Scope** — `Text`, `Vision`, `Audio`, or multiple.
3. **Provider** — e.g. `google`, `groq`, `openrouter`, `cerebras`, `ollama`.
4. **Internal ID** — e.g. `text_gemini_flash_lite`.
5. **API model name** (`full_name`) — e.g. `gemini-2.5-flash-lite`.
6. **Labels and quota** — Vietnamese/Korean/English names, quota strings, enabled state, and desired ordering.
7. **Default usage impact** — whether presets, fallbacks, or helper flows should start using the model.

## Steps

1. **Confirm the exact identifiers** before editing. Internal IDs and API model names are both used in this repo, and changing only one often leaves dangling references.

2. **Search the repo first**:
   ```
   rg -n "<internal-id>|<api-model-name>|supports_thinking|model_supports_search_by_name|BlockBuilder::text|get_model_by_id|GeneratedLiveModelCatalog|GeneratedPresetModelCatalogData" src mobile .claude scripts -S
   ```
   Also search for nearby sibling models from the same provider/family to copy the correct placement and behavior.

3. **Update the canonical manifest** in `catalog/model_catalog.json`.
   - Add, edit, or remove the relevant entry under `models`.
   - Update related sections when needed:
     - `constants`
     - `defaults`
     - `provider_defaults`
     - `priority_chains`
     - `realtime_transcription_aliases`
     - `tts_gemini_models`
     - `live_translation_providers`
   - Preserve ordering intentionally. Fallback selection and UI order still depend on catalog order.

4. **Regenerate derived catalogs**.
   - Rust reads the manifest through `build.rs`, so `cargo check` will regenerate the Rust catalog automatically.
   - Android preset/live catalogs are generated by `scripts/generate_android_preset_model_catalog.py`.
   - If you need to inspect the generated Kotlin outputs directly, use:
     ```
     python3 scripts/generate_android_preset_model_catalog.py --manifest-source catalog/model_catalog.json --preset-output /tmp/GeneratedPresetModelCatalogData.kt --live-output /tmp/GeneratedLiveModelCatalog.kt
     ```

5. **Audit capability rules and provider-specific logic**.
   Check whether the change requires updates in:
   - `catalog/model_catalog.json`
     - defaults, aliases, and capability-driving sections
   - `src/model_config.rs`
     - wrapper helpers such as `model_is_non_llm()`
     - `model_supports_search_by_name()`
   - `src/api/text/translate.rs`
   - `src/api/text/refine.rs`
   - `src/api/vision.rs`
   - `src/gui/settings_ui/help_assistant.rs`
   - Android runtime call sites that still contain provider-specific behavior
     - `mobile/androidApp/src/main/java/dev/screengoated/toolbox/mobile/service/`
     - `mobile/androidApp/src/main/java/dev/screengoated/toolbox/mobile/model/`

   Pay special attention to substring checks such as Gemini `supports_thinking` logic. New model families often need explicit handling even when the catalog entry itself is correct.

6. **Audit hardcoded internal ID consumers**.
   Search for the internal ID and update any default or fallback usage that should change:
   - `src/config/preset/defaults/text.rs`
   - `src/config/preset/defaults/image.rs`
   - `src/config/preset/defaults/audio.rs`
   - `src/api/text/refine.rs`
   - Any other `BlockBuilder::text("...")` or `get_model_by_id("...")` call sites found by `rg`

7. **Remove cleanly when deleting a model**.
   If the operation is `remove`, also remove or replace:
   - preset references
   - fallback defaults
   - helper/demo usages
   - capability exceptions tied only to the deleted model

8. **Verify with the repo-approved commands for WSL**:
   ```
   ORT_SKIP_DOWNLOAD=1 cargo check --target x86_64-pc-windows-gnu
   ./mobile/scripts/sgtp-wsl.sh gradle :androidApp:compileFullDebugKotlin --console=plain
   ```
   If working on native Windows and the environment supports it, also run:
   ```
   cargo clippy --all-targets
   ```

9. **Report what changed**.
   Summarize:
   - manifest changes
   - any generated-catalog impact
   - any capability-rule updates
   - any preset/default updates
   - verification status and any environment-specific limitations

## Key Facts

- The canonical model catalog now lives in `catalog/model_catalog.json`.
- Rust derives its catalog via `build.rs`; Android derives preset/live catalogs via `scripts/generate_android_preset_model_catalog.py`.
- Presets and workflow defaults still usually reference the internal model ID, not the API `full_name`.
- Search/grounding support is still inferred partly from `full_name` substring checks, so family-level naming changes can alter behavior unexpectedly.
- Gemini request behavior is not fully data-driven yet; some capabilities are still enabled by feature-specific logic in API call paths.
